# Tenjin visualizer scaffolding checklist

Current project shape:

```text
tenjin/
  src/                    React + Vite frontend
  crates/
    compiler_core/        migrated Rust compiler library
    compiler_wasm/        wasm-bindgen browser wrapper
  compiler/               old standalone compiler repo, kept as reference/native CLI
```

The old setup checklist is mostly complete. The next work is about turning the scaffold into a useful first compiler visualizer.

## Done

- Root Vite app exists.
- Root `package.json` has `dev`, `build:wasm`, `build`, `lint`, and `preview`.
- Root Cargo workspace exists.
- `compiler_core` contains the migrated compiler modules and tests.
- `compiler_wasm` builds through `wasm-pack`.
- `compiler_wasm::compile_source` calls into `compiler_core::lexer::lex`.
- `npm run build` produces a working static build.
- `cargo test -p compiler_core` passes.
- `cargo test -p compiler_wasm` passes.

## Next milestone: React -> WASM -> Rust -> JSON

Goal:

```text
source input -> compiler_wasm -> compiler_core -> JSON output panel
```

1. Add a frontend compiler API.

   Create `src/compilerApi.ts`:

   ```ts
   import init, { compile_source } from "./wasm/compiler_wasm/compiler_wasm";

   let initialized = false;

   export async function compileSource(source: string) {
     if (!initialized) {
       await init();
       initialized = true;
     }

     return JSON.parse(compile_source(source));
   }
   ```

2. Replace the default Vite UI.

   Build the first boring-but-real UI:

   - left panel: source textarea
   - button: run compiler
   - right panel: JSON output

   Start with plain textarea before CodeMirror.

3. Confirm this works in dev:

   ```sh
   npm run build:wasm
   npm run dev
   ```

   Milestone is complete when editing source and clicking run shows lexer token output from Rust.

## Improve the WASM result shape

Update `crates/compiler_wasm/src/lib.rs` so `compile_source` returns structured phases:

```json
{
  "ok": true,
  "diagnostics": [],
  "tokens": [],
  "parse": null,
  "semantic": null,
  "lowering": null,
  "assembly": null
}
```

Recommended order:

1. tokens
2. parse success/failure
3. semantic diagnostics
4. assembly text
5. lowering snapshots

Do not try to serialize the entire AST first. Start with strings and small view models, then add richer JSON once the UI knows what it wants.

## Add a Web Worker

## Add graph views later

Useful compiler hooks already exist:

- `compiler_core::ast::viz::program_to_dot`
- `compiler_core::lowering::pipeline::lowering_until`
- `LoweringPassKind`
- `LoweringSnapshot`

Browser graph path options:

1. Convert AST/lowering snapshots into JSON graph nodes/edges.
2. Render with React Flow / `@xyflow/react`.
3. Keep Graphviz PNG generation native-only, because browser/WASM should not shell out to `dot`.

Do not make graph visualization the next milestone. Get text/json phase inspection working first.

Add deploy scripts only after the visualizer has a useful first screen.
